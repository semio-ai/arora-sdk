# Cargo-first build migration

## Goal

Move the Engine repo from a CMake-orchestrated build (which then invokes cargo)
to a Cargo-orchestrated build (which invokes cmake/cc only where genuinely
needed for C++ pieces). The end state should let any downstream user pick a
target via standard cargo mechanisms (`--target ...`, `.cargo/config.toml`) and
get a working build without ever invoking CMake at the top level.

The repo must still build for three targets:

1. **host** (Mac arm64 / Linux x86_64)
2. **wasm32-wasip1** (or `wasm32-unknown-unknown` where currently used) via the
   WASI SDK
3. **NAO** = `i686-unknown-linux-musl`, using the Homebrew cross-toolchain on
   Mac

## Current state (snapshot, 2026-05-22)

- Top-level `CMakeLists.txt` orchestrates:
  - `cargo build` for host tools (`arora-cli`, `arora-module-cli`, etc.)
  - `cargo build --target wasm32-unknown-unknown` for `arora-buffers` and
    `arora-util` (used by C++ WASM modules)
  - `cargo build --target i686-unknown-linux-musl` when `NAO=ON`
  - `add_subdirectory(libs)` and `add_subdirectory(modules)` for C++ pieces
- `PreLoad.cmake` downloads WASI SDK 25 via `FetchContent` and sets it as the
  toolchain file.
- `modules/nao/CMakeLists.txt` fetches `libqi` from GitHub via `FetchContent` —
  this is the slow path the user wants stubbed.
- `modules/nao/build.rs` already calls back into cmake from cargo (inverse of
  the top-level pattern); good reference for the new world.
- C++ modules under `modules/` (test-cpp, test-cpp-2, behavior-tree-nodes,
  polly, nao) each have their own `CMakeLists.txt` and use code generated by
  the host tool `arora-module-cli` from a `module.yaml`.
- The C++ WASM modules produce *executables* with custom link flags
  (`--no-entry`, `--export=arora_buffer_alloc`, `--export=arora_buffer_free`) —
  these flags must survive the migration.

## Strategy

Staged migration, in this order:

### Stage 1 — Stub libqi (unblock iteration)

- Create a stub libqi (minimal C++ headers, no implementations) inside the
  repo. Functions either don't exist or trap when called. The goal is link, not
  run.
- Wire `modules/nao/CMakeLists.txt` to consume the stub instead of
  `FetchContent`-ing the real libqi from GitHub.
- Gate behind a feature/option so the real libqi can still be used when needed.

### Stage 2 — Cargo-first orchestration

- Promote `Cargo.toml` at the repo root as the source of truth (already a
  workspace; just expand its responsibility).
- Add `.cargo/config.toml` with:
  - `[target.i686-unknown-linux-musl]` block setting linker/ar/etc. (driven by
    env vars now, baked into the config)
  - `[target.wasm32-wasip1]` block (if we switch from `wasm32-unknown-unknown`)
- For each C++-containing module, replace its top-level `CMakeLists.txt`
  consumer-side with a `build.rs` that:
  - Locates the host tool `arora-module-cli` (built first by an
    `xtask`/Cargo dep, or shelled-out)
  - Generates the C++ bindings into `OUT_DIR`
  - Invokes `cmake::Config` (or `cc::Build` if simpler) to compile the C++
  - For WASM-executable modules: invokes the WASI SDK clang directly
- Delete the root `CMakeLists.txt` and `PreLoad.cmake` once equivalent
  functionality exists in Rust.

### Stage 3 — WASI SDK fetched from Rust

- Write a small helper (either an `xtask` or a `build-deps/` crate) that:
  - Downloads + caches WASI SDK 25 in `target/wasi-sdk-25/` on first use
  - Exposes paths to clang/lld/sysroot via env vars or a generated config
- Build scripts for WASM-targeted C++ modules consume those paths.

### Stage 4 — Tests

- Port `add_test` calls to either:
  - `tests/` integration tests under the appropriate crate (preferred where
    natural), or
  - An `xtask test` runner that knows how to invoke `arora-cli` against built
    module artifacts.

### Stage 5 — Validate three targets

In order: host → WASM → NAO. WASM is the riskiest because of the WASI SDK
ergonomics; NAO is annoying because it depends on Homebrew state.

## Decisions taken

- **WASM target:** standardize on `wasm32-wasip1` (Tier 2, well-supported, the
  default of `wasi-sdk` ≥ 31). Bump `wasi-sdk` 25 → **33** (latest, April 2026).
  Defer `wasm32-wasip2` (component model) to a future migration.
- **Host tools from cross-target builds:** use nightly `-Z bindeps`
  artifact-dependencies (RFC 3028, cargo issue #9096). Consumer crates declare
  the host tool with:
  ```toml
  [build-dependencies]
  arora-module-cli = { path = "...", artifact = "bin", target = "target" }
  ```
  Wait — `target = "target"` would build it for the *consumer's* target. To
  pin to host we use `target = "host"` (the RFC spelling) or omit `target`
  and rely on bindeps' default behavior for build-deps (which is host). The
  `build.rs` then reads `$CARGO_BIN_FILE_ARORA_MODULE_CLI` (cargo exports this
  for bindeps).
  Requires `cargo-features = ["bindeps"]` in the top-level `Cargo.toml` and a
  `rust-toolchain.toml` pinning nightly. Acceptable trade-off given how much
  simpler the consumer code becomes vs. recursive-cargo env-scrubbing.
- **xtask vs build.rs:** code generation per crate stays in `build.rs`. An
  `xtask` is added only if the test runner ends up needing more than `cargo
  test` can express.
- **Stub location:** `libs/qi-stub/` (sibling of `libs/cpp`). Header-only
  initially.

## Non-goals (for now)

- Replacing CMake *inside* individual C++ modules. As long as the orchestrator
  is Rust, modules can keep their own `CMakeLists.txt` invoked from `build.rs`.
- Reworking the module loader / VFS / runtime — pure build-system change.
